# 全量短信与高性能浏览设计

## 目标

在现有 Android uni-app 短信备份应用中取消全部广告与黑名单过滤，确保收件、发件、验证码、营销和服务通知等短信都会进入可靠队列并上传。同时把受保护短信浏览器升级为接近系统短信应用的会话体验：使用真实联系人备注、手机号和头像，完整展示收发方向、正文与元数据，并通过原生分页、渐进加载和窗口化图片网格保证大量短信与一万张图片场景仍可流畅操作。

本轮只完成设计、实现、自动测试和 App 资源构建，不提交 DCloud 云打包，也不生成新的 APK。

## 已确认需求

- 取消所有短信过滤。淘宝、拼多多、验证码、广告、退订等内容全部保存并上传。
- 已被旧版本过滤的历史短信必须在升级后重新扫描并补入上传队列。
- 会话列表、会话详情使用同一份联系人身份信息，不再用名字或号码的首字作为头像。
- 联系人身份包含备注名、原始手机号、号码类型标签和真实联系人缩略头像。
- 没有联系人照片或没有联系人权限时使用统一的默认联系人图片资源，并保留完整手机号或短信地址。
- 会话详情完整展示收到与发出的短信、联系人备注、手机号、正文、时间、方向、状态、SIM、彩信附件等已有数据。
- “发送短信”指展示系统中已经发送的短信，不增加主动发送短信功能。
- 短信和图片都不能一次性读取、解析和渲染全部数据。
- 一万张图片场景只允许加载视口附近的缩略图，不允许创建一万个图片节点或同时解码原图。
- 新增与性能相关的关键代码必须使用清晰中文注释，说明边界、原因和降级策略；不写重复代码含义的逐行注释。

## 产品体验

### 会话列表

入口保持现有十分钟密码会话。解锁后默认进入“会话”，顶部显示剩余时间、刷新和锁定操作。会话行采用固定高度，包含真实头像或默认头像、联系人备注、完整手机号、最后一条消息预览、最近时间、消息数量和未读数量。联系人备注与手机号分两行展示，服务号或无联系人记录的地址不伪造成联系人名称。

会话列表先返回轻量汇总，不把每条短信正文都传到前端。Android 原生层在 IO 线程扫描 SMS/MMS 必要列并按 `thread_id` 聚合，只把会话摘要传回页面。扫描期间页面显示明确的读取状态，但主线程保持可点击和可滚动。

### 全部、收到与已发送

三个消息页采用基于时间和消息 ID 的游标分页，每页最多 40 条。第一次只读取首屏，滚动接近底部再取下一页。每条消息显示联系人备注、手机号、收发方向、完整正文、准确时间、消息类型、已读状态、发送状态、SIM 和短信中心等可用信息。

页面只保存已加载的有限批次；超过窗口上限时回收远离当前位置的渲染节点。已读取的数据可以留在十分钟内存会话中用于返回定位，但不能把所有消息同时挂到 DOM。

### 会话详情

点击会话后按 `thread_id` 分页查询 SMS 和 MMS，再按统一毫秒时间合并。收到的消息使用左侧气泡，发出的消息使用右侧气泡；顶部联系人区域复用会话列表中的头像、备注、手机号和号码类型。每个气泡保留完整正文，底部显示方向、时间、状态、SIM 等信息。彩信附件只渲染缩略图，点击后才打开原图预览。

### 图片与相册

图片页先读取相册元数据与数量，不解码图片。当前相册按页读取 60 条 URI，并使用固定单元格尺寸的虚拟网格：页面只创建视口上下缓冲区内约 30 至 45 个图片节点。图片页缓存最多四个数据页，离开缓存窗口后释放对应组件与 URI 引用。

缩略图组件启用延迟加载并关闭不必要的重复动画；只有点击图片时才把当前图片及相邻预览 URI 交给全屏预览。不能为了预览一张图片预先构造一万条 `urls` 数组。全屏预览只加载当前页及相邻页，用户继续翻页时再补充数据。

## 权限与联系人身份

Android Manifest 新增 `READ_CONTACTS`。解锁短信浏览器后单独请求联系人权限，不把联系人权限与短信备份授权绑定。用户拒绝联系人权限时，短信读取、备份和上传仍然工作，页面显示完整手机号与默认头像。

联系人解析器在 IO 线程读取 `ContactsContract.CommonDataKinds.Phone` 的号码、标准化号码、显示名称、号码类型、自定义标签和 `PHOTO_THUMBNAIL_URI`。号码通过 Android `PhoneNumberUtils` 标准化，结果按标准化号码缓存，避免对每条短信单独查询联系人 Provider。联系人照片使用系统返回的缩略图 URI；没有照片时使用项目内的默认联系人 PNG 资源。

联系人数据只保存在当前十分钟查看会话的内存中，不写入应用数据库，不进入短信上传请求，也不发送到 Node 服务端。

## 原生数据接口

新增分页接口，所有 Provider 查询继续在 `UTSAndroid.getDispatcher('io')` 中执行：

- `getConversationSummaries(password)`：返回聚合后的会话摘要和联系人身份，不返回完整历史正文数组。
- `getMessagePage(password, filter, threadId, cursor, limit)`：按全部、收到、已发送或指定线程返回最多 40 条 SMS/MMS，并提供下一页游标和 `hasMore`。
- `getGalleryAlbums(password)`：返回相册 ID、名称和图片数量，不返回图片内容。
- `getGalleryPage(password, albumId, offset, limit)`：返回最多 60 条图片元数据、总数和是否还有下一页。
- `requestContactsPermission()`：只请求联系人读取权限。

Android 8 及以上优先使用结构化查询参数 `QUERY_ARG_LIMIT`、`QUERY_ARG_OFFSET` 和排序参数；如果厂商 Provider 未声明或未正确支持分页参数，降级为受控 Cursor 跳过与截取，仍然只构建一页 JSON，不构建全量 JSON 数组。

短信与彩信合并使用稳定游标：`timestamp`、`kind`、`sourceId` 共同确定下一页边界。相同时间的消息不会重复或遗漏。原生响应必须限制 `limit` 的合法范围，短信最大 100、图片最大 120，避免页面错误参数造成大结果集。

## 取消过滤与数据迁移

`SmsRepository.queueRecord` 不再调用 `SmsFilter.match`，所有不重复的短信直接进入 SQLite 队列。首页删除“已过滤”统计和黑名单入口，删除黑名单页面、规则存储、原生规则保存接口及相关测试，避免保留已经失效的业务入口。

SQLite 数据库版本升级。升级时清空或删除旧 `filtered_records` 数据，并让去重查询只检查 `sms_queue`，确保旧版本曾被过滤的短信在下一次历史扫描时能够重新入队。上传协议与 Node 服务端保持不变，仍使用 `recordId` 幂等去重。

## 前端状态与文件边界

- `sms-viewer-session.ts`：只负责十分钟密码会话、联系人身份缓存、会话摘要和有限分页缓存。
- `sms-viewer-pagination.ts`：负责短信游标页合并、去重、窗口截取和加载状态。
- `virtual-media-grid.ts`：负责根据滚动位置计算可见图片范围、需要的页码和四页 LRU 缓存。
- `messages.vue`：负责会话、消息分类与图片分页的页面编排，不直接实现分页算法。
- `conversation.vue`：负责指定会话的气泡渲染和按页加载，不复制主页面的联系人解析逻辑。
- `SmsRepository.kt`：负责 Provider 查询、联系人解析、分页与数据库入队，不负责页面颜色和文案。

复杂逻辑前使用中文块注释解释为什么需要分页、如何避免重复、为什么限制缓存；简单赋值、模板字段和明显样式不添加噪声注释。

## 视觉系统

继续使用现有 Material 3 风格的紫色主色、浅灰背景、圆角 Surface 和绿色发件气泡。调整为一致的 8rpx 间距基线、统一圆角和清晰的文字层级。会话头像固定尺寸，姓名、手机号、预览和时间使用稳定对齐；加载状态不造成列表跳动。

默认头像必须是实际 PNG 资源，不使用字符、Emoji、CSS 图形或名字首字替代。页面中的真实照片统一使用圆形裁切，会话详情复用相同 URI，避免同一联系人前后头像不一致。

## 性能约束

- ContentProvider 查询不得在 Android 主线程执行。
- 首屏短信正文最多 40 条，单次原生短信页最多 100 条。
- 单次相册页默认 60 条，前端同时存在的图片节点目标不超过 45 个。
- 图片数据页缓存最多四页；切换相册、锁定或会话到期立即释放。
- 会话列表只传摘要，不能附带完整消息历史。
- 页面计算不得在每次滚动时重新遍历全部短信或全部图片。
- 滚动事件使用帧级节流，只在可见行范围或页边界变化时更新响应式状态。
- 大图只在用户点击预览时读取，网格不主动解码原始尺寸图片。

## 错误与降级

- 联系人权限拒绝：显示号码和默认头像，短信功能不受影响。
- 图片权限拒绝：会话与短信继续可用，只隐藏相册照片；MMS 附件仍按短信权限读取。
- 厂商不支持结构化分页：使用 Cursor 截取兼容路径，并保持响应条数上限。
- 分页期间会话到期：丢弃返回数据、清空缓存并回到密码页。
- 图片加载失败：显示默认图片失败状态，不重复无限请求，不影响其他图片滚动。
- 页面快速切换筛选或相册：旧请求带有请求代次，返回后发现代次过期则丢弃，防止旧数据覆盖新页面。

## 测试与验收

先写失败测试，再实现每个行为：

- 取消过滤后任意广告或营销正文都进入队列；旧过滤记录不再阻止重新扫描。
- 联系人号码标准化、备注、类型和头像映射一致；拒绝权限时正确降级。
- 短信游标分页在相同时间戳下不重复、不遗漏，并严格限制页大小。
- 会话摘要不包含完整消息数组，会话详情按页合并 SMS/MMS。
- 虚拟图片网格在一万条元数据下只返回视口附近索引，LRU 缓存不超过四页。
- 页面契约覆盖联系人头像、备注、手机号、完整正文、收发方向、加载更多和图片虚拟网格。
- 原生契约覆盖 `READ_CONTACTS`、分页参数、Provider 查询、权限降级和删除过滤调用。
- 运行 `npm test -- --run`、`npm run type-check` 和 `npm run build:app-plus`。
- 不执行 HBuilderX 云打包。真机性能仍需后续在至少一台包含大量短信和大量图片的 Android 设备上验证。
